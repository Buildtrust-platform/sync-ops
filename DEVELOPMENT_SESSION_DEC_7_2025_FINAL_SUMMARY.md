### Database Schema Design

**CallSheet Model (Main)**
```typescript
{
  // Production Info
  productionTitle, productionCompany, shootDayNumber,
  totalShootDays, shootDate, episodeNumber,

  // Call Times
  generalCrewCall, estimatedWrap, timezone,

  // Location
  primaryLocation, primaryLocationAddress, parkingInstructions,
  nearestHospital, hospitalAddress,

  // Weather
  weatherForecast, temperature, sunset,

  // Contacts
  director, producer, firstAD, productionManager (names + phones),

  // Additional
  mealTimes, cateringLocation, transportationNotes,
  safetyNotes, specialInstructions, nextDaySchedule,

  // Metadata
  status, version, publishedAt, lastUpdatedBy,

  // Relationships
  scenes[], castMembers[], crewMembers[]
}
```

**Key Design Decisions:**
1. **Time Storage:** String format (HH:mm) for timezone flexibility
2. **Model Separation:** Normalized schema for scenes/cast/crew
3. **Status Workflow:** DRAFT ‚Üí PUBLISHED ‚Üí UPDATED ‚Üí CANCELLED
4. **Version Tracking:** Auto-increment for audit trail

## üéì LESSONS LEARNED

### What Worked Well
1. **Comprehensive Planning** - Implementation plan guided entire session
2. **Normalized Schema** - Separate models for scenes/cast/crew scales better
3. **Component Separation** - Form and Viewer as separate components
4. **Industry Research** - Web search for call sheet standards was invaluable
5. **Progressive Disclosure** - Build core first, enhancements later
6. **Documentation First** - Plan before code prevented rework

### Challenges Encountered
1. **Form Complexity** - Large form required careful organization
2. **Time Format Decision** - String vs datetime required thought
3. **Type Safety** - Some manual type assertions needed with Amplify types
4. **Relationship Handling** - Understanding Amplify's belongsTo/hasMany

### Improvements for Next Session
1. **Form Builder Library** - Consider using react-hook-form to reduce boilerplate
2. **Component Library** - Reusable form field components
3. **Auto-save** - Debounced saves for better UX
4. **Optimistic Updates** - Update UI before server confirmation
5. **Error Boundaries** - Better error handling at component level
